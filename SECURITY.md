# Security Policy

## Статус проекта

**vls** — учебный проект, не предназначенный для использования в production. Никаких гарантий целостности данных, защиты от concurrent-доступа или устойчивости к атакам не даётся. Не клади в `vls`-репозиторий ничего, что не готов потерять.

## Поддерживаемые версии

| Версия        | Поддержка                                 |
| ------------- | ----------------------------------------- |
| `main`        | да, фиксы выпускаются по мере возможности |
| любая прошлая | нет, обновись до `main`                   |

## Куда сообщать об уязвимостях

**Не открывай публичный issue для security-проблем.** Отправь подробности на почту мейнтейнера: **issue@oriplal.dev** или **rinat@gazizullin31.ru**.

В письме желательно указать:

- описание уязвимости и оцениваемое воздействие;
- минимальный воспроизводящий пример (последовательность команд / содержимое репо);
- версию проекта (commit-хеш или тег);
- окружение: ОС, компилятор, версии OpenSSL и cJSON.

Ответим в течение нескольких дней. Если уязвимость подтвердится — обсудим план фикса и сроки публичного раскрытия.

## Известные ограничения по безопасности

Перечисление того, что в текущей версии **не защищено** — чтобы не репортили заведомое:

- **Path traversal в `commit.json`.** При `checkout`/`reset` пути из commit.json используются как есть. Если репозиторий содержит запись с путём вида `../../etc/passwd`, запись в working tree пойдёт по этому пути. Не клонируй и не разворачивай чужие vls-репозитории, которым не доверяешь.
- **Коллизии MD5.** Контент-адресация построена на MD5; криптографически он сломан. Для учебной задачи этого достаточно, но как защиту от целенаправленной подмены blob'ов это не рассматривай.
- **Отсутствие блокировок.** Параллельный запуск двух `vls` в одном репозитории может привести к гонкам и битому состоянию. Не запускай конкурентно.
- **Нет проверки целостности objects.** Хранимые blob'ы не верифицируются по хешу при чтении — повреждение файла на диске остаётся незамеченным до явного перехеширования.

## Что в области security и интересует

Сообщения о следующем — приветствуются:

- buffer overflow / out-of-bounds в C-коде (особенно вокруг `vls_join_path`, `mmap`-чтения, ручного парсинга);
- утечки памяти / fd, обнаруженные ASan/Valgrind;
- символлинк-атаки внутри `.vls/`;
- любые случаи, когда `vls` пишет за пределы `.vls/` или working tree без явной команды пользователя.
