# Velosiped System

> "Не такие метели в лицо летели" — когда Git кажется слишком тяжелым, а писать на Bash — западло.

**vls** — это хардкорная система контроля версий, написанная на чистом **C23** с использованием **POSIX** системных вызовов. Здесь нет места высокоуровневой магии, только `open`, `read`, `write` и голая арифметика битов.

## Технологический стек

*   **Язык:** C23 (только хардрок)
*   **Стандарт:** POSIX (никаких Windows-костылей, только POSIX)
*   **Криптография:** OpenSSL EVP API для MD5 хеширования
*   **Хранилище:** Самописное хранилище с адресацией по контенту
*   **Сборка:** CMake

## Сборка

Зависимости: `cmake ≥ 3.21`, `gcc ≥ 14` или `clang ≥ 18`, `openssl`, `cjson`.

```sh
cmake -S . -B build
cmake --build build
```

Бинарь окажется в `build/vls`. Тесты запускаются через `ctest`:

```sh
ctest --test-dir build --output-on-failure
```

## Как это устроено внутри

Вся твоя история живёт в скрытой папке `.vls`:

*   `objects/` — Хранилище файлов. Именем файла является его MD5-хеш. Если контент одинаковый — в базе одна копия.
*   `commits/<hash>/` — Каталог одного коммита. Внутри:
    *   `commit.json` — снимок stage.json в момент коммита (массив `{path, hash, status}`),
    *   `hash` — собственный хеш коммита (содержимое stage.json в этот момент),
    *   `msg` — сообщение коммита,
    *   `prev` — хеш родителя (пусто для самого первого коммита),
    *   `time` — когда был создан.
*   `stage.json` — Опись проекта. JSON-массив `{path, hash, status}` для каждого подготовленного к коммиту файла.
*   `head` — Указатель на текущую вершину истории (хеш последнего коммита).

История — это односвязный список: каждый коммит знает только своего родителя через `prev`. Чтобы пройти историю, идёшь от `head` назад по цепочке.

## Команды

### Инициализация `init`

Инициализирует логово зверя. Создает структуру папок, необходимых для работы.
```bash
vls init
```

### Отслеживание `add`

Берёт твои файлы или целые папки, хеширует их содержимое и кладёт копии в `objects/`, фиксируя пути в `stage.json`.
```bash
vls add <files>
```

### Состояние `status`

Показывает, что сейчас в стейдже и какие файлы изменены.
```bash
vls status
```

### Сохранение `commit`

Фиксирует текущее состояние проекта. Создаёт каталог `commits/<hash>/` со всеми метаданными, обновляет `head` и сбрасывает статусы в стейдже на UNCHANGED.
```bash
vls commit "сообщение коммита"
```

### Возвращение `reset`

Откатывает состояние проекта на указанный коммит: восстанавливает файлы из `objects/` в working tree, перезаписывает stage снапшотом этого коммита и двигает `head`. «Лишние» файлы (которых нет в target-коммите) остаются на диске как есть — это упрощённая семантика, не подчищаем за пределами index'а.
```bash
vls reset <hash>
```

### Дроп `drop`

Убирает файл из стейджа, не трогая working tree. Аналог `git rm --cached`. Полезно, если случайно проиндексировал что-то и не хочешь коммитить.
```bash
vls drop <files>
```

### История `log`

Выводит всю историю проекта от HEAD до самого первого коммита.
```bash
vls log
```

## Быстрый старт

```sh
cd /tmp/my-project
vls init
echo "первая строчка" > readme.txt
vls add readme.txt
vls commit "первый коммит"
vls log
```

## Что работает, что нет

- ✅ `init`, `add`, `commit`, `log`, `status`, `reset`, `drop` — полный цикл индексации, истории и отката.
- ✅ Удаление файлов: если ты `rm`-нул файл и сделал `commit`, он автоматически отметится как DELETED в снапшоте и пропадёт из stage. Отдельного `vls rm` нет — реконсиляция со working tree происходит внутри `commit`.
- ✅ Хранилище `objects/` — append-only, content-addressed: старые версии файлов не затираются, `reset` на любой прошлый коммит честно восстанавливает то состояние.
- ⚠️ `reset` не подчищает «лишние» файлы в working tree (overwrite-only семантика).
